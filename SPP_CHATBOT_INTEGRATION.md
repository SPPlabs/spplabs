# SPP Labs AI Chatbot Integration Specification

This document serves as the official specification and drops-in instructions for integrating the AI Chatbot into client websites generated by SPP Labs. When generating a new client website, developers and AI agents must follow this document to implement the chatbot.

---

## 1. Architecture

Client websites managed by SPP Labs are independent Next.js applications. The chatbot UI and routing proxy must reside natively inside each client website's own codebase.

The connection flow is as follows:

```
Visitor Browser (React UI)
        |
        | [POST /api/chat { message }]
        v
Client Website Next.js Server Route (API Proxy)
        |
        | [POST https://api.spplabs.es/api/chat]
        | [Authorization: Bearer SPP_AI_API_KEY]
        | [Body: { website_id: SPP_WEBSITE_ID, message }]
        v
Central SPP Labs API Gateway (api.spplabs.es)
        |
        v
AI Services (Qdrant Retrieval -> vLLM Qwen3 completions)
```

No direct browser communication with the central API gateway is permitted.

---

## 2. Security Rules

To protect the client's account and the central AI infrastructure from unauthorized access, the following rules are mandatory:

*   **NEVER** expose the `SPP_AI_API_KEY` in client-side React components, browser logs, or browser JavaScript bundles.
*   **NEVER** prefix the API key or website ID environment variables with `NEXT_PUBLIC_`. Doing so bundles them into the client-side JavaScript sent to the browser.
*   **NEVER** permit the browser to call `https://api.spplabs.es/api/chat` directly.
*   **NEVER** commit plain-text API keys or client configuration files containing secrets to version control repositories.
*   The browser must communicate **only** with the website's own server-side proxy route (`/api/chat`). The website's server acts as the secure gateway that injects credentials.

---

## 3. Environment Variables

Each generated client website must include the following server-only variables in its `.env` configuration file:

```env
# The registered domain name of the client website (acts as website_id)
SPP_WEBSITE_ID="client-domain.com"

# The private API key issued by SPP Labs for this website
SPP_AI_API_KEY="spp_api_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

*Note: In local development, these variables should be placed in `.env.local`. In production, configure them securely in your server environment variables panel (Vercel, Docker Env, etc.).*

---

## 4. API Contract

### A. Client Website Internal Route
*   **Path:** `POST /api/chat`
*   **Request Payload (JSON):**
    ```json
    {
      "message": "What are your business hours?"
    }
    ```

### B. Proxy Forwarding Request
The internal route must parse the incoming message, inject environment variables, and make a server-to-server request:
*   **URL:** `POST https://api.spplabs.es/api/chat`
*   **Headers:**
    ```http
    Authorization: Bearer SPP_AI_API_KEY
    Content-Type: application/json
    ```
*   **Payload (JSON):**
    ```json
    {
      "website_id": "client-domain.com",
      "message": "What are your business hours?"
    }
    ```

### C. Streaming Response
*   The central API returns a `text/event-stream` response.
*   The client website proxy route must pipe/forward this stream directly back to the browser.
*   The frontend client must handle progressive rendering, reading chunks dynamically from the response body stream reader (e.g. using `TextDecoder` and `ReadableStreamDefaultReader`).

---

## 5. Website Implementation Instructions

When building a chatbot for a generated client website, follow these guidelines:

1.  **Native Components:** Build the chatbot UI natively within the client website's codebase (e.g. `src/components/chatbot/`).
2.  **Branding and Design System:** Match the exact branding, typography, color palette, and component design system of the parent website (e.g. using tailwind classes configured in the site's `tailwind.config.js`).
3.  **Minimal Dependencies:** Rely on native web APIs (like `fetch` and standard Web Streams) to implement streaming responses. Do not install heavy external chatbot client libraries.
4.  **Graceful Error Handling:** If the API proxy returns an error, catch it and show a friendly support message (e.g., "Sorry, I'm having trouble connecting. Please try again later.") rather than dumping trace stack logs or raw network errors to the visitor.

---

## 6. Do Not Implement

The following architectures are explicitly out of scope for client website integrations:

*   **Do not create script tag injection widgets** (like `<script src="https://.../widget.js">`).
*   **Do not generate public chatbot tokens** for the browser.
*   **Do not share frontend libraries** or build shared npm packages across different client websites.
*   Each website receives its own local component design, ensuring complete code independence and freedom for custom design modifications.

---

## 7. Future Compatibility

Future updates to the central SPP Labs gateway may introduce:
*   API client rate limiting
*   Token usage limits per website
*   Dashboard analytics charts
*   Dynamic layout customization payloads

All such extensions will be managed transparently at the central API level. The client website integration contract (`POST /api/chat` server-proxy calling `api.spplabs.es/api/chat`) will remain backward-compatible and unchanged.
